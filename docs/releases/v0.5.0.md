# AgentForge v0.5.0 — Application

> Released 2026-06-16. Coordinated release train (ADR-0015): every
> workspace package bumps to `0.5.0` in lockstep.

This train is the first shaped almost entirely by **dogfooding** — a
sibling agent built on `agentforge-py 0.2.4` filed the gaps it hit, and
every one of them became a fix or an extension point here. The theme:
making the framework genuinely *configurable* and *hostable* by a real
application, and closing the config→runtime seams that let a few of
those gaps hide from line coverage.

---

## Highlights

- **Application config (`app:` + sections + `imports:`)** — `agentforge.yaml`
  now has a reserved `app:` namespace for your own config, optional
  registered+validated sections, and an `imports:` directive to split
  config across files. The framework stores/validates it; your agent
  reads it with `AgentForgeConfig.app_as(MyModel)`. (feat-026)

- **Claude on AWS Bedrock, by config** — `providers.<name>.config` is now
  passed through to the provider constructor (it was documented but
  dropped), and the Bedrock provider gained STS `role_arn` assume-role.
  An AWS agent runs the Agent runtime + `BudgetPolicy` on Bedrock with one
  IAM credential set, no boto3 glue. (enh-004)

  ```yaml
  providers:
    default:
      type: bedrock
      model: "us.anthropic.claude-haiku-4-5-20251001-v1:0"
      config: { region: us-east-1, role_arn: "arn:aws:iam::…:role/bedrock" }
  ```

- **Auth in front of MCP HTTP** — `MCPServer.from_http(..., middleware=[…])`
  is the seam for a bearer-token gate (or CORS / rate-limit) on the
  streamable-HTTP transport, without re-implementing the serve path. (enh-003)

- **uv-native `add module`** — `agentforge add module` now uses an
  environment-aware installer (`uv add` in a uv project, else pip), so the
  uv-first scaffold's documented next step works and persists. (bug-021)

- **Hardened config→runtime path** — `modules.memory` builds from config
  for every backend, `agentforge run --replay` reconstructs the strategy,
  and a new offline configured-agent E2E (per-PR **and** against the
  release wheel) guards the whole seam. (bug-022, bug-023)

## What's new

### Added

- **feat-026 — application config extension.** Reserved top-level `app:`
  block + `AgentForgeConfig.app_as(model, key=None)`; registered typed
  sections via the `agentforge.config_sections` entry-point group
  (validated in `agentforge config validate`); and an `imports:` directive
  for multi-file config (Spring `spring.config.import` precedence, `${ENV}`
  + overlay + `--resolved` all apply). See
  [feat-026](../features/feat-026-application-config-extension.md), ADR-0022.
- **enh-004 — provider `config:` passthrough + Bedrock assume-role.**
  `providers.<name>.config` reaches the provider constructor (`region` /
  `aws_profile` / `role_arn` / `timeout_seconds`); Bedrock chat + embedding
  clients gain `role_arn`/`role_session_name` STS assume-role. Runbook 13
  documents the Bedrock path (inference-profile ids, IAM). See
  [enh-004](../enhancements/enh-004-bedrock-provider-config-and-assume-role.md).
- **enh-003 — MCP HTTP middleware seam.** `MCPServer.from_http(middleware=…)`.
  See [enh-003](../enhancements/enh-003-mcp-http-middleware-seam.md).

### Fixed

- **bug-021** — `agentforge add module` / `remove module` use an
  environment-aware installer (uv project → `uv add`/`uv remove`; classic
  venv → `python -m pip`; else `uv pip`), instead of a `python -m pip` that
  is absent in a uv-managed venv.
- **bug-022** — `modules.memory` builds from config for every backend
  (sqlite/postgres/neo4j/surrealdb): each store now implements the async
  `from_config` factory the build path expects; `build_memory_from_config`
  is async. Previously any configured memory backend failed at the CLI.
- **bug-023** — `agentforge run --replay` reconstructs the agent with its
  configured strategy (and budget / system prompt / iteration cap) and
  closes the agent after a run, so a configured backend doesn't leak.

### Changed

- **Test/release hardening (no API change).** A new offline
  configured-agent E2E (build → run a real agent with `modules.memory` +
  `providers.config` + replay) runs on every PR and against the built wheel
  in the pre-publish gate, so a documented config knob that stops reaching
  the runtime fails CI.

### Security

- None.

## Breaking changes

**None** for the public locked contracts (the `agentforge-core` ABCs are
unchanged; all additions are additive). Two internal CLI helpers changed
signature and are **not** part of any locked contract — listed only for
contributors:

- `agentforge.cli._build.build_memory_from_config` is now `async`.
- The memory `MemoryStore` drivers gained an async `from_config` factory
  (additive on the concrete drivers, not the ABC).

A consumer using the public surface (`Agent`, `agentforge.yaml`, the CLI)
upgrades with no code changes.

## Install / upgrade

```bash
# New install
pip install "agentforge-py[anthropic]==0.5.0"   # or [bedrock], [openai], …

# Upgrade
pip install --upgrade "agentforge-py==0.5.0"
```

For a project scaffolded with an earlier `agentforge new`:

```bash
agentforge upgrade --to 0.5.0
```
