# AgentForge v0.2.1 — Publishable

The first PyPI-published AgentForge release. **Same shipped
surface as v0.2.0; only the distribution name and a few
metadata pins changed.**

v0.2.0 was tagged on 2026-05-14 but never uploaded to PyPI
because the bare name **`agentforge`** on PyPI is owned by an
unrelated project ([`DataBassGit/AgentForge`](https://github.com/DataBassGit/AgentForge)
v0.6.5). v0.2.1 renames the runtime distribution to
**`agentforge-py`**, pins every cross-package dependency
to `~= 0.2.1`, and wires PyPI Trusted Publishing — so this tag
push uploads all 34 workspace packages to PyPI under the
`scaffoldic` account with no long-lived API token.

---

## Highlights

- **`pip install` is now real.** Fresh installs go through PyPI:

  ```bash
  pip install "agentforge-py[anthropic]==0.2.1"
  agentforge new my-agent --template minimal
  ```

  All 34 workspace packages publish in lockstep via the new
  `release.yml` Trusted Publishing workflow.

- **Python import name unchanged.** `from agentforge import
  Agent` continues to work — only the PyPI distribution name
  changed (`agentforge` → `agentforge-py`). Mirrors how
  `python-dateutil` ships (installs as `python-dateutil`,
  imports as `dateutil`).

- **Cross-package deps pinned.** Every workspace member
  declares `agentforge-* ~= 0.2.1`. ADR-0015's
  coordinated-release-train invariant now holds for installed
  wheels: a user installing `agentforge-anthropic==0.2.1`
  cannot pair it with a future incompatible
  `agentforge-core==0.5.0`.

- **No functional changes.** Same surface as v0.2.0 — 5 LLM
  providers, 4 rerankers, 4 observability backends, 4 vector
  stores, schema migrations, GraphRAG, sentence-window
  guardrails, MCP / A2A production runners, etc. See the
  [v0.2.0 notes](./v0.2.0.md) for the feature catalogue.

## What's new

### Changed

- **Distribution rename: `agentforge` → `agentforge-py`.**
  - `packages/agentforge/pyproject.toml` — `name = "agentforge-py"`.
  - Root `pyproject.toml` — workspace `dependencies` and
    `[tool.uv.sources]` switched to `agentforge-py`.
  - Sister packages that declared a runtime dependency
    (`agentforge-chat`, `agentforge-chat-http`,
    `agentforge-a2a`, `agentforge-testing`) now depend on
    `"agentforge-py ~= 0.2.1"`.
  - The Python import name stays `agentforge` —
    `[tool.hatch.build.targets.wheel] packages = ["src/agentforge"]`
    is unchanged. End-user code is **unaffected**.

- **Cross-package deps pinned to `~= 0.2.1`.** Affects every
  `agentforge-*` reference across all 34 `pyproject.toml`
  files. Previously bare (e.g. `"agentforge-core"`), now
  bounded to `>= 0.2.1, < 0.3.0`.

- **All 34 workspace members bumped to `0.2.1`.** Every
  `__version__` constant updated to match.

### Added

- **`.github/workflows/release.yml`** — PyPI release workflow.
  Triggers on `push: tags: ["v*"]` and `workflow_dispatch`.
  Two-job pipeline:
  1. **`build`** — `uv build --all`, upload `dist/` artefacts.
  2. **`publish`** — gated on the `pypi` GitHub environment
     (manual approve), uses `pypa/gh-action-pypi-publish`.
     **Hybrid auth:** the workflow uses the
     `PYPI_API_TOKEN` GitHub secret today; conversion to
     Trusted Publishing (OIDC) is deferred and tracked in
     [`playbooks/publish-to-pypi.md`](../../playbooks/publish-to-pypi.md)
     §0a. `skip-existing: true` so re-running on a partial
     failure is safe.

### Removed

- The transitive resolution of `agentforge==0.6.5` (the
  squatted upstream project) from `uv.lock`. The renamed
  workspace dependency now resolves to the local
  `agentforge-py==0.2.1` member.

## Breaking changes

**One distribution-level rename.** Users who installed
`agentforge` from the GitHub source pre-v0.2.1 should switch:

```diff
- pip install agentforge[anthropic]
+ pip install agentforge-py[anthropic]
```

Python imports do not change:

```python
from agentforge import Agent     # still works
from agentforge.testing import MockLLMClient  # still works
```

Scaffolded agents from `agentforge new` keep working; their
`pyproject.toml` should pin `agentforge-py ~= 0.2.1`.

---

## Migration guide

### If you installed from GitHub source

```bash
pip uninstall agentforge
pip install "agentforge-py[anthropic]==0.2.1"
```

### If you have a scaffolded agent on v0.2.0

```bash
agentforge upgrade --to 0.2.1
```

The upgrade hook rewrites `pyproject.toml` to depend on
`agentforge-py ~= 0.2.1`. Your custom code stays untouched.

### If you publish your own sister package

Update your `pyproject.toml`:

```diff
 dependencies = [
-    "agentforge-core",
-    "agentforge",
+    "agentforge-core ~= 0.2.1",
+    "agentforge-py ~= 0.2.1",
 ]
```

## Coordinated release train

Per ADR-0015, every framework release bumps every in-scope
workspace package to the same minor version simultaneously.
v0.2.1 ships **34 packages** — same set as v0.2.0, each at
`0.2.1` with the new pinned cross-package deps. The renamed
runtime package is `agentforge-py 0.2.1`; everything else
keeps its name and just bumps version.

## Install / upgrade

```bash
# Fresh install
pip install "agentforge-py[anthropic]==0.2.1"

# From a scaffold
pip install agentforge-py==0.2.1
agentforge new my-agent --template minimal
cd my-agent && agentforge run "Hi"

# Upgrade an existing scaffolded agent
agentforge upgrade --to 0.2.1
```
